## blunt assessment

**short version:** yes, this can be a good bachelor thesis, but **not** if it is presented mainly as "an e-commerce app with AI".

that version is too close to:

- product prototyping
- frontend/backend integration work
- API orchestration
- "LLM wrapper" work

those things are useful, but by themselves they are usually **not enough** to feel like a strong computer science thesis in 2026.

**it becomes thesis-worthy if you frame it as:**

> **design, implementation, and evaluation of a trustworthy, reactive-first AI shopping advisor that combines classical recommendation methods, behavior signals, and LLM-generated explanations**

that framing is much better because it turns the work into a CS problem about:

- system architecture
- recommender design
- human-AI interaction
- trust calibration
- evaluation methodology
- performance/cost tradeoffs

---

## the real thesis

the app should be the **artifact**, not the thesis itself.

the thesis should be about a concrete technical question such as:

1. **Does a reactive-first advisor outperform a proactive one in user trust and perceived usefulness?**
2. **Does combining classical recommenders with an LLM explanation layer produce better user outcomes than either one alone?**
3. **Can lightweight behavior signals improve the timing and relevance of shopping assistance without becoming intrusive?**
4. **What architecture gives the best tradeoff between relevance, latency, cost, and user trust?**

if you can answer even one of those convincingly, the project becomes much stronger academically.

## recommended thesis framing

### strongest framing for the current project

**working thesis title options:**

1. `A Reactive-First AI Shopping Advisor: Combining Recommender Systems, Behavior Signals, and LLM Explanations`
2. `Design and Evaluation of a Trustworthy AI Shopping Assistant for E-Commerce`
3. `Behavior-Aware Explainable Recommendations with Large Language Models in E-Commerce`

### thesis statement

the thesis is not "can i build a store with AI".

the thesis is:

> how to design and evaluate an AI-assisted shopping system that is useful, non-intrusive, explainable, and technically efficient.

### research questions

recommended final research questions:

1. **RQ1:** does a reactive-first advisor produce higher user trust and lower perceived intrusiveness than proactive auto-triggered suggestions?
2. **RQ2:** does an LLM explanation layer improve the perceived usefulness of recommendations generated by classical algorithms?
3. **RQ3:** which behavior signals are most useful for determining readiness for assistance?
4. **RQ4:** what are the latency and cost tradeoffs of a two-stage recommendation architecture?

### expected contributions

for a bachelor thesis, these are realistic and solid:

1. a working end-to-end prototype
2. a hybrid recommendation architecture
3. a behavior-driven readiness model
4. an evaluation of trust, usefulness, latency, and cost
5. design guidelines for building non-intrusive AI shopping assistance

## what makes it worthy for computer science

to make this feel like **computer science**, not just app development, you should emphasize:

### 1. system design

- why the architecture is split into static recommendation + LLM explanation
- why reactive-first interaction was chosen
- why behavior signals are aggregated client-side
- why the AI should not be called on every interaction

### 2. algorithmic reasoning

- co-occurrence with Jaccard
- content-based similarity
- hybrid recommendation logic
- ranking and reranking strategy
- evaluation metrics for recommendation quality

### 3. human-computer interaction

- trust
- explainability
- timing of interventions
- proactive vs reactive interaction
- perceived intrusiveness

### 4. empirical evaluation

- offline recommender metrics
- system metrics
- user study or structured usability evaluation

### 5. engineering tradeoffs

- speed vs quality
- cost vs usefulness
- reactivity vs architectural complexity
- privacy vs personalization

## what is not enough

if the final result looks like any of the following, it will feel weak:

- a polished storefront plus a chatbot
- a demo with no measurements
- a lot of implementation but no research question
- generic LLM usage with no comparison or justification
- lots of product features but no real evaluation
- strong UI/UX but weak technical analysis

the easiest failure mode is this:

> you spend months building the store and only a few days evaluating it.

that produces a good portfolio project, but a mediocre thesis.

## evaluation plan

this is the most important section if you want the thesis to feel serious.

### 1. offline recommender evaluation

evaluate the classical and hybrid recommendation quality with:

- `Hit Rate@5`
- `NDCG@10`
- catalog coverage
- diversity
- popularity bias

this proves you did not just "make recommendations" — you assessed them.

### 2. system evaluation

measure:

- median and p95 latency for recommendation queries
- median and p95 latency for LLM advice
- token usage per advice request
- estimated cost per session / per 100 users
- cache hit impact if you enable prompt caching

this proves architectural awareness.

### 3. user study or structured usability evaluation

for a bachelor thesis, you do **not** need a huge formal study. a realistic version is enough:

- `8-15` participants
- task-based comparison
- each participant tries:
  - baseline store without AI
  - store with reactive-first advisor
- collect:
  - perceived usefulness
  - trust
  - intrusiveness
  - clarity of explanations
  - purchase confidence

good instruments:

- short Likert-scale questionnaire
- `SUS` for usability if you want a standard measure
- trust/adoption questions adapted from AI trust literature

### 4. strongest comparison to run

if time allows, compare:

1. **baseline recommender only**
2. **recommender + proactive AI**
3. **recommender + reactive-first AI**

if time is tighter, compare:

1. **baseline recommender only**
2. **recommender + reactive-first AI**

that is already enough for a bachelor thesis.

## best way to present it to a thesis committee

the committee should leave thinking:

> this student did not just build a demo. they studied a relevant modern problem, designed a system around it, and evaluated it.

### presentation structure

1. **problem**
   - shoppers are overwhelmed
   - generic AI chatbots are often untrusted and intrusive
   - recommendation systems are useful but opaque

2. **research gap**
   - current AI assistants often optimize for "smartness" instead of trust and timing
   - there is room for a system that is recommendation-driven, explainable, and reactive-first

3. **proposed solution**
   - hybrid architecture
   - behavior-aware readiness signals
   - user-initiated LLM advice

4. **system design**
   - architecture diagram
   - algorithms
   - AI flow
   - evaluation setup

5. **results**
   - recommendation quality
   - trust/usability results
   - latency/cost results

6. **discussion**
   - what worked
   - where the AI helped
   - where it did not
   - limitations

### demo strategy

demo this exact story:

1. baseline product discovery
2. classical recommendation slots
3. subtle advisor readiness signals
4. user opens advisor
5. advisor gives concise, explainable suggestions
6. show measured latency and evaluation summary

the demo should reinforce the thesis claim, not just show features.

## brutally honest recommendation

if you keep the current idea, **do not** present it as:

> an AI e-commerce store

present it as:

> a trustworthy human-centered AI recommendation and decision-support system for e-commerce

that sounds more serious because it is more serious.

the modern academic angle is **not** "I used AI".

the modern academic angle is:

- when should AI intervene?
- how should AI justify itself?
- when does AI help vs annoy?
- what hybrid architecture is actually efficient and trustworthy?

that is what feels current in 2026.
